# Next.js（App Router）× 完全CSR向け 4層アーキテクチャ方針

## 前提

- 本番では Next サーバーは起動せず、`output: 'export'` で静的出力した成果物を Spring Boot から配信する（SSG＋クライアント）。 [nextjs](https://nextjs.org/docs/app/guides/static-exports)
- `src/app` 以下の `.tsx` は **すべて Client Component として扱う**（`"use client"` 前提）。  
- エラーは「継続不能エラー」と「継続可能エラー」を明確に分けて扱う。  

用語：

- **UIコンポーネント**: 純粋に描画だけを行うコンポーネント。  
- **フィルターコンポーネント**: Provider / ErrorBoundary / Guard など、ロジック（初期化・権限・エラーハンドリング）を担当するコンポーネント。

***

## 分類する2種類のエラー

### 継続不能エラー

- このセッションを継続できないエラー。  
- ケース例:
  - アプリ全体の初期化通信（マスターデータ、ログインユーザ情報、共通権限など）の失敗（ステータスコード問わず）。  
  - 画面固有の初期化時に発生した 401 / 403（セッション切れ・権限なし）。  
- 振る舞い:
  - 必要に応じてログアウトの API を呼ぶ（この通信が失敗してもよい）。  
  - デザインレイアウトを適用せず、「致命的エラー画面」を表示。  
  - ユーザには「ウィンドウを閉じてログインしなおしてください」と案内。  

### 継続可能エラー

- この画面の表示は成立していないが、アプリ全体としては継続させたいエラー。  
- ケース例:
  - 画面レンダリング中の API エラー（401 / 403 を除くあらゆるエラー: 4xx, 5xx, ネットワークエラーなど）。  
- 振る舞い:
  - 通常のデザインレイアウト（ヘッダ／フッタ）を適用した状態で、画面全体を覆うエラーUIを表示（画面の中身は描画しない）。  
  - ステータスコードに応じたメッセージ＋「リトライ」「前の画面に戻る」ボタンを表示。  
  - next のクライアントルーターでは遷移ガードがないため、「初期データが取れずこの画面を表示すべきでない場合」も、この全画面エラーでカバーし、リトライ／戻るで対応する。

***

## 4層アーキテクチャの構造

### 第1層: インフラ層（ほぼ落ちないフィルター）

**責務**

- HTML骨組みとグローバルな Provider の定義。  
- 通信や権限チェックなど「落ちやすい処理」は書かない。  

**構成例**

```txt
app/
  layout.tsx         ← HTML, body, グローバルProvider（QueryClientProvider 等）
  global-error.tsx   ← root layout 自身の致命的エラー時の最終保険（本番専用）
```

- `app/layout.tsx`:  
  - `<html lang="ja">` / `<body>` の定義。  
  - 各種 Provider（React Query, 状態管理, テーマなど）。  
  - 「同期的で失敗しにくい初期化」以外は書かない。  
- `app/global-error.tsx`:  
  - `app/layout.tsx` 自体や root 周りで発生した致命的エラーの最終表示。  
  - dev では Next の開発用エラーオーバーレイが優先されるため、本番専用の最終砦とみなす。 [github](https://github.com/vercel/next.js/issues/55462)

***

### 第2層: 継続不能エラー層（アプリ基盤フィルター）

**責務**

- アプリ全体の初期化と「セッション継続不能」の判定。  
- ここで起きるエラーは「このセッションは終了」扱いとする。  

**構成例**

```txt
app/
  (root)/
    layout.tsx   ← AppBootstrapper, 共通権限チェックなど
    error.tsx    ← 継続不能エラー画面（ログアウト＋再ログイン案内）
```

**典型的な処理内容**

- `app/(root)/layout.tsx`:
  - `AppBootstrapper` を配置し、  
    - アプリ初期化通信（マスターデータ、ログインユーザ、共通権限）  
    - 401 / 403 を含む初期化段階のエラー  
    をまとめて扱う。  
  - AppBootstrapper 内の Query は **`throwOnError: true`** とし、  
    - エラーをローカルの ErrorBoundary で飲み込まず、  
    - `(root)/error.tsx` までそのまま伝播させる（継続不能エラー）。  

- `app/(root)/error.tsx`:
  - デザインレイアウトを使わない「致命的エラー画面」。  
  - 必要に応じてログアウトの API を呼び出し、ユーザに再ログインを促す。

***

### 第3層: 継続可能エラー層（共通レイアウト＋共通エラー画面）

**責務**

- アプリの基本デザイン（ヘッダ／フッタなど）を適用するレイヤー。  
- 画面単位では継続不能でないが、「この画面の中身は表示しないほうがよい」エラーを全画面エラーUIとして扱う。  
- 第3層共通の `error.tsx` で「リトライ可能な全画面エラー」を提供する。

**構成例**

```txt
app/
  (root)/
    (app)/
      layout.tsx   ← デザインレイアウト＋画面共通フィルター
      error.tsx    ← 継続可能エラー画面（レイアウト適用＋リトライ／戻るボタン）
      screen1/
        page.tsx
      screen2/
        page.tsx
```

**役割**

- `app/(root)/(app)/layout.tsx`:
  - デザイン上のレイアウト（ヘッダ／フッタ／共通 PermissionGate など）を配置。  
  - 下位の page が例外を投げた場合、第3層の `error.tsx` で「レイアウト付きエラー画面」を出す。  

- `app/(root)/(app)/error.tsx`:
  - 全画面で共通の「継続可能エラー画面」。  
  - ステータスコードに応じたメッセージ＋「リトライ」「前の画面に戻る」を提供。  
  - レイアウト（ヘッダ／フッタ）を適用した状態で、コンテンツ領域だけをエラーUIで埋めるイメージ。  

***

### 第4層: 個々の画面（page）層

**責務**

- 画面固有の UI とロジック。  
- error.tsx は置かず、例外はすべて上位（第3層や第2層）のハンドラに任せる。

**構成例**

```txt
app/
  (root)/
    (app)/
      screen1/
        page.tsx
      screen2/
        page.tsx
```

**エラーハンドリングの基本方針**

- 画面内で使う TanStack Query などは、  
  - 401 / 403 → 「継続不能エラー」として明示的に throw（第2層に昇格させる）。  
  - それ以外のエラー → 「継続可能エラー」として第3層の error.tsx または個別 ErrorBoundary で扱う。  
- `page.tsx` 自体には `error.tsx` を置かず、  
  - 「この画面だけで完結するエラーUI」が必要な場合は、  
    - `QueryErrorResetBoundary + ErrorBoundary` を組み合わせたローカルなフィルターコンポーネントを使い、  
    - 第3層のレイアウト内で「コンテンツだけエラーUIに差し替える」。

***

## まとめ

- **第1層:** 落ちにくいインフラ層（HTML＋グローバルProvider＋global-error）。  
- **第2層:** アプリ全体の初期化・ログイン・共通権限など、「セッション継続不能」を扱う層（AppBootstrapper＋(root)/error.tsx）。  
- **第3層:** デザインレイアウト＋継続可能エラー用の共通 error.tsx（リトライ／戻る付き全画面エラー）。  
- **第4層:** 各画面（page）。ここには error.tsx を置かず、例外はすべて上位のフィルターに任せる。

この4層を意識することで、

- UIコンポーネントとロジックコンポーネント（フィルター）を分離しやすくなり、  
- 継続不能／継続可能エラーの責務を階層で整理でき、  
- 別アプリでも再利用可能な Next.js 用フレームワークとして構築しやすくなります。
