# Next.js App Router レイヤー構造メモ

このドキュメントは、現在の App Router 構成における各 `layout.tsx` / `error.tsx` の責務と、Route Group の命名意図を整理するためのメモである。Next.js の Route Groups は URL に影響せず、レイアウトや責務ごとにルートを整理する用途に使えるため、この構成では画面見た目ではなく**責務の境界**でグループを分ける方針を採用している。[1][2]

## 構成イメージ

```txt
app/
│  error.tsx
│  global-error.tsx
│  layout.tsx
│
└─(recoverable)
    │  error.tsx
    │
    └─(bootstrap)
        │  layout.tsx
        │
        └─(shell)
            │  layout.tsx
            │
            ├─detail
            │      page.tsx
            ├─results
            │      page.tsx
            └─search
                   page.tsx
```

## 命名方針

この構成では、Route Group 名を「画面の種類」ではなく「その階層が何を保証し、何を受け持つか」で決める。Route Groups は URL には出ないため、エンドユーザー向け名称ではなく、開発者が見て責務をすぐ理解できる名前を優先してよい。[1][3]

採用候補は次の 3 つである。

- `(recoverable)` — リトライ可能なエラー境界を持つ層。
- `(bootstrap)` — アプリ起動時の初期化処理を担当する層。
- `(shell)` — ヘッダやフッタなど、アプリの見た目の枠を提供する層。

## app/layout.tsx

`app/layout.tsx` は**最上位の絶対に落としたくないレイヤー**として扱う。ここには、通信・状態依存・権限依存などの失敗要因を持ち込まず、HTML の土台や最低限の静的ラッパーなど「失敗しないこと」を優先したコードのみを置くのがよい。

このレイヤーが落ちると `app/error.tsx` や `app/global-error.tsx` に流れるが、ここでのエラーは通常のリトライ対象ではなく、**致命的エラー**として `/error` などの専用エラーページへ退避させる前提にする。

## app/error.tsx と app/global-error.tsx

この 2 つは、**アプリ継続が難しい致命的エラー用**の境界である。ここに来た場合は部分復旧を目指すのではなく、アプリ全体として安全側に倒し、`window.location.replace('/error')` のように専用エラーページへ遷移させる扱いが分かりやすい。[4][5]

`error.tsx` は該当セグメント配下のエラー、`global-error.tsx` はより広い範囲の致命的失敗に対応する。どちらも「復旧 UI」ではなく「致命扱いの退避経路」と割り切っておくと、下位の recoverable な層との責務が混ざりにくい。

## (recoverable)/error.tsx

`(recoverable)` は、**リトライ可能なエラーを受け止める層**である。この配下の `error.tsx` では再試行ボタンを表示し、`reset()` を使って再レンダーを試みる。[4][6]

ただし、すべてのエラーをここで吸収するわけではない。401 や 403 のように権限・認証の問題であり、ここで再試行しても意味が薄いものは再スローして、上位の致命エラー層または専用制御に委ねる方が責務として自然である。

## (bootstrap)/layout.tsx

`(bootstrap)` は、**アプリを利用可能な状態まで初期化する責務**を持つ層である。ここには通信を伴う Provider、たとえば PermissionProvider、Profile の取得、AppData の取得、初回起動時のガード処理などを配置する。

このレイヤーで重要なのは、「アプリ共通で必要な初期データの取得失敗」と「画面個別データの取得失敗」を分けることである。Profile や AppData の取得に失敗した場合は、再試行前に store を `blank` に戻してからエラーを再スローすることで、下位の recoverable error 境界の `reset()` によって bootstrap をやり直せる構成にしておくのが扱いやすい。[7][4]

## (shell)/layout.tsx

`(shell)` は、**アプリ全体の見た目の枠**を提供する層である。ここにはヘッダ、フッタ、ナビゲーション、コンテンツ領域のレイアウトなど、デザイン上の共通枠を置く。

この層の役割は「起動準備」ではなく「描画の器」を提供することなので、`(bootstrap)` と分離しておく意味が大きい。これにより、「初期化に失敗したのか」「見た目の枠を描いているのか」がディレクトリ名だけで把握しやすくなる。

## ページ配下

`detail`、`results`、`search` などの `page.tsx` は、**実際の画面機能**を持つ層である。ここでは画面固有の検索条件、詳細取得、一覧取得などを扱うが、それらの失敗は bootstrap 失敗とは別物として考える。

たとえば通信状態の悪化により、ある画面の初期データ取得だけ失敗することはあり得る。この場合は AppData や Profile をリセットせず、画面単位の recoverable error として `(recoverable)/error.tsx` の retry で再試行させる設計が妥当である。[5][4]

## この構造の意図

このレイヤー構造の意図は、エラーを「全部同じもの」として扱わず、**どこで発生した失敗か**で処理方針を分けることにある。特に、bootstrap 失敗時だけ store の reset が必要であり、画面固有エラーではそれを避けたいという要件に対して、責務を階層で分ける構成は相性がよい。

また、`(base)` や `(pages)` のような広い名前より、`(recoverable)`、`(bootstrap)`、`(shell)` のように意図を含む名前の方が、後から見返したときに設計の理由が残りやすい。Route Groups はプロジェクト整理のための機能なので、命名も構造説明の一部として扱うのがよい。[1][2]

## 運用ルール案

運用時の判断をぶらさないために、次のルールを置いておくとよい。

- `app/layout.tsx` には失敗要因を持ち込まない。
- `app/error.tsx` / `app/global-error.tsx` は致命的エラー専用にする。
- `(recoverable)/error.tsx` は retry 可能なエラーだけを扱う。
- `(bootstrap)/layout.tsx` は起動時に必須な通信・Provider だけを担当する。
- `(shell)/layout.tsx` は見た目の枠に集中し、起動責務を持たせない。
- 画面固有通信の失敗では AppData / Profile をリセットしない。
- AppData / Profile の取得失敗時のみ、reset 後に再スローして recoverable error 境界へ渡す。

このルールを明文化しておくと、今後レイアウトやエラー境界を追加するときにも責務がぶれにくい。
